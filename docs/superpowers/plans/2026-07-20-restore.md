# Restore Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let the user pick a backup ZIP and replace the current library with its contents, without ever destroying data that has not first been copied and proven restorable.

**Architecture:** Extends `BackupManager`, which already owns tool discovery, archive verification and process handling. Restore validates the archive, takes a safety backup, trial-loads the dump into a scratch database, and only then drops and reloads the real one.

**Tech Stack:** C++17, Qt 6.11.1, QML, PostgreSQL 16, `pg_dump`, `psql`, `/usr/bin/zip`, `/usr/bin/unzip`.

**Spec:** [2026-07-20-restore-design.md](../specs/2026-07-20-restore-design.md)

**Branch:** `feature/restore` (already checked out).

## Read This First — This Plan Can Destroy Real Data

Every other feature in this project was additive. This one drops a schema.

The database `wormbook` holds the user's real library: 89 books, tags, a reading
session, challenges. **It is not test data and there is no other copy.**

Rules for every task here:

1. **Never run `DROP` against `wormbook` while developing.** Use a scratch database.
2. **Before any task that could touch `wormbook`, take a real backup first** and report
   its path in your findings.
3. Test the destructive path against a scratch database created from a dump, never
   against the live one.
4. If a step would have you verify by wiping the user's library, stop and report
   instead. A missing verification is recoverable; their data is not.

A scratch database is created like this:
```bash
createdb wormbook_dev_scratch
psql -q -d wormbook_dev_scratch -f /path/to/database.sql
# ... test against wormbook_dev_scratch ...
dropdb wormbook_dev_scratch
```

    Q_INVOKABLE QVariantMap inspectArchive(const QString &filePath);

- [ ] **Step 2: Implement it**

```cpp
QVariantMap BackupManager::inspectArchive(const QString &filePath)
{
    QVariantMap result;
    result["valid"]     = false;
    result["bookCount"] = 0;
    result["hasCovers"] = false;

    QString path = filePath;
    if (path.startsWith(QStringLiteral("file://")))
        path = QUrl(path).toLocalFile();

    if (m_psqlPath.isEmpty()) {
        result["error"] = QStringLiteral("psql not found — restore unavailable");
        return result;
    }

    // 1. The archive opens and holds what it must.
    QProcess list;
    list.start(QStringLiteral("/usr/bin/unzip"), {QStringLiteral("-l"), path});
    if (!list.waitForFinished(30000) || list.exitCode() != 0) {
        result["error"] = QStringLiteral("File is not a readable ZIP archive");
        return result;
    }

    const QString listing = QString::fromUtf8(list.readAllStandardOutput());
    if (!listing.contains(QStringLiteral("database.sql"))) {
        result["error"] = QStringLiteral("Archive does not contain database.sql");
        return result;
    }
    result["hasCovers"] = listing.contains(QStringLiteral("covers/"));

    // 2. Unpack to a temporary directory.
    QTemporaryDir temp;
    if (!temp.isValid()) {
        result["error"] = QStringLiteral("Could not create a temporary directory");
        return result;
    }

    QString unpackError;
    if (!runProcess(QStringLiteral("/usr/bin/unzip"),
                    {QStringLiteral("-q"), QStringLiteral("-o"), path,
                     QStringLiteral("-d"), temp.path()},
                    &unpackError)) {
        result["error"] = QStringLiteral("Archive could not be unpacked");
        return result;
    }

    // 3. Trial load into a scratch database. This is the moment a corrupt dump is
    //    caught, and it happens while the real database is still untouched.
    const QString scratch = QStringLiteral("wormbook_restore_check");
    dropScratchDatabase(scratch);   // in case a previous run died mid-way

    QString error;
    if (!runProcess(m_createdbPath, {scratch}, &error)) {
        result["error"] = QStringLiteral("Could not create a scratch database: ") + error;
        return result;
    }

    const bool loaded = runProcess(
        m_psqlPath,
        {QStringLiteral("--quiet"),
         QStringLiteral("--set=ON_ERROR_STOP=1"),
         QStringLiteral("--dbname=") + scratch,
         QStringLiteral("--file=") + temp.filePath(QStringLiteral("database.sql"))},
        &error);

    if (!loaded) {
        dropScratchDatabase(scratch);
        result["error"] = QStringLiteral("Archive could not be loaded: ") + error;
        return result;
    }

    // 4. Count what arrived.
    result["bookCount"] = scratchBookCount(scratch);
    dropScratchDatabase(scratch);
    result["valid"] = true;
    result["error"] = QString();
    return result;
}
```

This references three helpers you must also add: `m_createdbPath` (discovered exactly
like `psql`, for the `createdb` binary), `dropScratchDatabase(name)` (runs `dropdb
--if-exists`), and `scratchBookCount(name)` (opens a short-lived `QSqlDatabase`
connection named something other than the app's default connection, runs
`SELECT count(*) FROM books`, closes and removes it).

**The connection-name detail matters:** opening a `QSqlDatabase` with the default
connection name would displace the application's live connection. Use
`QSqlDatabase::addDatabase("QPSQL", "restore_check")` and
`QSqlDatabase::removeDatabase("restore_check")` when done. Read how
`DatabaseManager::connect()` sets its connection up and mirror the parameters from
`BookWorm::Config`.

### Task 3: The restore itself

**Files:**
- Modify: `src/backup/backupmanager.h`
- Modify: `src/backup/backupmanager.cpp`

- [ ] **Step 1: Declare it**

```cpp
    // Replaces the current database with the archive's contents. Takes a safety
    // backup first and refuses to proceed if that fails.
    Q_INVOKABLE bool restoreFrom(const QString &filePath);

signals:
    void restoreFinished(bool ok, const QString &message);
```

- [ ] **Step 2: Implement it**

Order matters. Write it in exactly this sequence:

1. Resolve the path, strip `file://`.
2. Bail out if `m_psqlPath` is empty.
3. **Safety backup first.** Call the existing `backupTo()` with
   `safetyBackupDir() + "/before-restore-<timestamp>.zip"`. If it returns false, emit
   `restoreFinished(false, ...)` and return — do not proceed. Keep the path in a local;
   every later failure message must include it.
4. Unpack the archive into a `QTemporaryDir`.
5. Drop and recreate the schema:
   ```cpp
   runProcess(m_psqlPath,
              {QStringLiteral("--quiet"),
               QStringLiteral("--set=ON_ERROR_STOP=1"),
               QStringLiteral("--dbname=") + QString::fromLatin1(BookWorm::Config::DB_NAME),
               QStringLiteral("--command=DROP SCHEMA public CASCADE; CREATE SCHEMA public;")},
              &error);
   ```
6. Load the dump with `--set=ON_ERROR_STOP=1`. **If this fails, the message must lead
   with the safety backup path** — this is the one window where data is genuinely at
   risk.
7. Restore covers (Step 3 below).
8. Emit `restoreFinished(true, ...)` with the number of books restored.

The application holds an open `QSqlDatabase` connection throughout. After the schema is
replaced underneath it, that connection's cached state is stale. Read
`DatabaseManager` and decide the honest handling: either reconnect, or document clearly
that `initializeSchema()` plus a reload is enough. **Test whichever you choose against a
scratch database and report what you observed** — do not assume.

- [ ] **Step 3: Restore covers**

Add a private helper that, given the unpacked directory:
- creates `<AppDataLocation>/covers`
- copies each `covers/<id>.<ext>` there
- runs `UPDATE books SET cover_image_path = :path WHERE id = :id` for each

Covers absent from the archive keep whatever path the dump restored.

Failure here must NOT abort: the data is already restored. Count the failures and
report them.

- [ ] **Step 4: Run migrations after loading**

An archive predating a table or column must arrive fully upgraded. After the dump
loads, call `DatabaseManager::instance().initializeSchema()`.

Confirm by reading `initializeSchema()` that it is safe to call on a populated
database — it should be, since it runs on every launch, but verify rather than assume.
